# 手順書化（skill オーサリング）

「設計書を作って」「〇〇の仕様を教えて」のように**繰り返されるユーザー依頼**を、毎回ゼロから対応すると品質・手順がぶれ、やり方を都度聞き直して人間同士のコミュニケーションが増える。このスキルは、そうした依頼から**手順を抽出して再利用可能な手順書 skill（`.claude/skills/<name>/SKILL.md`）として切り出す** authoring 手順を定義する。

controller が手動で起動する軽量・自己完結の補助スキル。**本スキルは「skill を作る」ところまでを担い、手順そのものの実行は生成された skill の役割**とする。

## 解決する問題

- **毎回ゼロから対応し品質がぶれる**: 同じ依頼でも実行者・タイミングで手順や成果物の質が揺れる。
- **やり方が人の頭にしか無く渡せない**: 暗黙のやり方が形式知化されておらず、他のエージェント／人に引き継げない。
- **やり方を都度聞き直す**: 入力・出力・品質基準を毎回確認し、人間同士のコミュニケーションが増える。

## いつ起動するか（トリガー）

- **明示依頼**: ユーザーが「この手順を skill 化して」「手順書にして」と言ったとき。
- **検知提案**: controller が同種依頼の繰り返しを観測したら、「手順書化しますか?」と提案し、**承認を得てから**起動する。検知の手がかり:
  - 同一セッション内で同種の依頼が目安 2〜3 回以上現れた。
  - `$memory-recall` で `.docs/memory` を想起し、同種の過去 work item／判断が見つかった。

### 手順化適性の判定基準

次を満たすものだけ skill 化する（満たさなければ作らず、理由を伝えて no-op とする）:

- 繰り返し性がある（一度きりでない）。
- 手順が安定している（毎回ほぼ同じ流れで進む）。
- 入力と出力が明確（何を受け取り何を返すかが言える）。
- 自動化価値がある（形式知化で品質・速度が上がる）。

毎回人間の裁量が支配的で手順が定まらないものは skill 化に向かない。

## 基本ルール

- 出力は**単一の `.claude/skills/<name>/SKILL.md`**。既存 skill の見出し集合（概要／解決する問題／基本ルール／手順／停止条件／関連）に合わせる。
- **ユーザーの「やり方の好み・品質基準・禁止事項・テンプレ」を必ず取り込む**。不明点は推測で埋めず質問する。
- **既存 skill と重複しないか確認**してから作る。重複するなら新規作成せず既存 skill を拡張する。
- 軽量・自己完結を保つ。work item 記録・`$empirical-prompt-tuning` 検証は任意（規模が大きい／重要なときに推奨）。
- 本スキルは skill 定義ファイルを書くのみ。生成 skill の手順を**この場で実行しない**。

## 手順（4 フェーズ）

```
CAPTURE → DRAFT → CHECK → REGISTER
```

### フェーズ1: CAPTURE — 依頼から手順を抽出する

- トリガー（明示依頼 or 検知提案の承認）を確認する。
- 上記「手順化適性の判定基準」で適性を判定する。向かなければここで no-op。
- `$memory-recall` で過去の同種依頼・実際にやった手順・ユーザーの判断/好みを想起する。
- 不足している要素をユーザーに質問して埋める:
  - **入力**: 何を受け取るか（前提・引数・参照先）。
  - **ステップ**: どの順で進めるか。
  - **出力形式**: 何をどんな形で返すか。
  - **品質基準・禁止事項**: 守るべき基準、やってはいけないこと。
  - **具体例**: 代表的な 1〜2 ケース。

### フェーズ2: DRAFT — SKILL.md を起草する

- 既存 skill の構成に合わせて `SKILL.md` を書く。
  - `name`: kebab-case。`.claude/skills/` 配下に同名が無いか**重複チェック**。
  - `description`: 「いつ／なぜ／何のときに使うか」を 1 段落で明示。
  - 本文: 概要／解決する問題／基本ルール／手順（ステップ or フェーズ）／出力フォーマット／停止条件／関連。
- CAPTURE で得たユーザーの基準・禁止事項・テンプレ・具体例を本文に落とす。
- 他 skill に依存する箇所は `$skill-name` 記法で参照する。

### フェーズ3: CHECK — 最小確認

軽量チェックのみ（本格検証は任意の REGISTER で）:

- **description ↔ body 整合**: `$empirical-prompt-tuning` の Iteration 0 相当を軽く行い、description が約束する内容を body が実際に提供しているか確認する。
- **手順の穴チェック**: CAPTURE の具体例 1 つで手順を机上でなぞり、入力から出力まで欠落なく辿れるか確認する。
- **停止条件・出力形式の明確さ**: 「いつ終わるか」「何を返すか」が曖昧でないか確認する。
- 要約（name／用途／手順の骨子）をユーザーに提示して確認を取る。

### フェーズ4: REGISTER — 確定と任意の記録

- skill を確定する。**新規 skill はセッション再読み込み（または次回起動）で有効になる**旨をユーザーに伝える。
- **任意**（規模が大きい／重要なとき推奨）:
  - `capability-expansion` / `skill-gap` ラベルの改善 work item を `$local-memory` で `.docs/memory` に記録する。
  - `$empirical-prompt-tuning` で本格的な品質検証（シナリオ反復）を行う。
  - 改善 work item として継続管理が必要なら `$loop-engineering` の振り返り、工程ゲートは `$cross-review-gates`／`verifier` に接続する。

## 生成する SKILL.md の雛形

```md
---
name: <kebab-case-name>
description: <いつ／なぜ／何のときに使うかを 1 段落で>
---

# <タイトル>

## 概要
<目的・位置づけ・スコープ（やること／やらないこと）>

## 解決する問題
- <従来の失敗・課題1>
- <従来の失敗・課題2>

## 基本ルール
- <守るべき制約・前提>

## 手順（ステップ or フェーズ）
1. <ステップ1: 入力と出力>
2. <ステップ2>
...

## 出力フォーマット
<返す成果物の形・テンプレート>

## 停止条件
- <完了条件>
- <エスカレーション／no-op 条件>

## 関連
- `$<skill>` — <役割と使い分け>
```

## 停止条件

- 新しい skill が `.claude/skills/<name>/SKILL.md` として作成され、最小確認（CHECK）を通過し、ユーザーが確認した。
- または、手順化適性なしと判定し、skill を作らず理由を伝えた（no-op）。

無理に skill 化を進めない。適性が無い・手順が定まらない場合は作らないことを選ぶ。

## 関連

- `$memory-recall` — CAPTURE で過去の同種依頼・判断/好みを `.docs/memory` から想起する。
- `$empirical-prompt-tuning` — REGISTER での任意の本格品質検証（重要／大規模な手順書のとき）。
- `$local-memory` — REGISTER での任意の work item 記録（`capability-expansion`/`skill-gap`）のフォーマット。
- `$loop-engineering` / `$cross-review-gates` — 改善 work item として継続管理・工程ゲートに接続する場合の参照先。
