# メモリ想起パターン

`.docs/memory` には work item の意図・実装・レビュー観点・人間の判断/フィードバックが蓄積されている。だがどこに何があるかは暗黙知で、問いに答えるには複数ファイルを横断して読む必要がある。このスキルは、**問い（なぜ / 何を / どう見るか）から該当記録を段階的に特定し、出典引用付きで合成して答える**読み取り手順を定義する。

検索の回し方は `$iterative-retrieval`（DISPATCH→EVALUATE→REFINE→LOOP）を土台にし、対象フォーマットの定義は `$local-memory` を参照する。

## 解決する問題

- **意図が分からず人に聞く**: 「なぜこの実装なのか」が記録にあるのに、辿り方が分からず人間に確認してしまう。
- **全文を読んでコンテキストが膨れる**: 答えは 1 エントリで足りるのに work item を丸ごと読み込み、コンテキスト枠を浪費する。
- **どこに何があるか分からない**: 意図・実装・レビュー観点・判断がそれぞれ別セクションにあることを知らず、空振りする。

## 基本ルール

- **読み取り専用**: `.docs/memory` を変更しない。引き出した内容は回答（提案）として返すだけ。
- **出典引用必須**: すべての回答に出典を付ける。粒度は `ファイルパス + セクション見出し or エントリ id`（例: `issues/0001-...md ## 人間の判断・フィードバック [判断]05`）。
- **記録が無ければ「記録なし」**: 該当する記録が見つからない場合は、推測で補わず「記録なし」と正直に返す。
- **推測しない**: メモリに書かれていないことを断定しない。書かれていない範囲は「未記録」と区別する。

## 検索マップ（取り出す情報 → 置き場所）

work item（`.docs/memory/issues/<id>-*.md`）と横断フィードバック（`.docs/memory/feedback/`）のどこに何があるかの対応表。

| 取り出す情報 | 置き場所 |
|---|---|
| 意図・背景（なぜ） | `## 目的` / `## 背景` / `## 人間の判断・フィードバック` の `### [判断]` |
| 実装内容（何を） | `## 受け入れ条件` / `## 実行ログ` の `### エージェント実行` |
| レビュー観点（基準と結果） | `## レビュー観点`（基準）＋ `## 実行ログ` の `### 工程レビュー`（結果） |
| 状態遷移の理由 | `## 実行ログ` の `### 状態遷移` / frontmatter `status`・`labels` |
| 人間のフィードバック | `## 人間の判断・フィードバック` の `### [フィードバック]` |
| 横断的な判断・規約 | `.docs/memory/feedback/`（`## F<NN>`、`出典 work item` で逆引き） |

## 標準手順（4 フェーズ）

`$iterative-retrieval` の DISPATCH→EVALUATE→REFINE→LOOP を、メモリ想起向けに具体化する。

### フェーズ1: LOCATE — 候補を索引で特定

問いに関係しそうな work item / フィードバックを索引から絞る。

- `.docs/memory/index.md`（work item 進捗ボード）で id / title / status を見て候補 work item を選ぶ。
- 横断的な規約・判断が問われている場合は `.docs/memory/feedback/README.md`（scope 別索引）で `F<NN>` を選ぶ。

出力: 候補 work item / フィードバックエントリのリスト。

### フェーズ2: GREP — タグで該当箇所を抽出

候補を全文読みする前に、タグで必要な箇所だけを特定する。`Grep` で使う代表パターン:

```
### \[判断\]           # 人間の判断（論点・選択肢・決定・理由）
### \[フィードバック\]  # 人間のフィードバック
## レビュー観点          # 確認基準
### 工程レビュー         # レビュー結果（基準との照合）
### 状態遷移            # status 変更とその理由
^## F[0-9]             # 横断フィードバックエントリ
^status:  / ^labels:   # frontmatter での絞り込み
出典 work item / 関連改善 work item   # トレーサビリティの逆引き
```

ヒットが多すぎる / 用語が外れて空振りする場合は REFINE（キーワードを実物の用語へ置換、対象パスを絞る）して再 GREP する。

### フェーズ3: READ & SYNTHESIZE — 読んで問いに答える

絞った箇所を `Read` で開き、問い（なぜ / 何を / どう）に対する答えを組み立てる。

- 方針転換の経緯を問われたら、`### [判断]` の連鎖や `出典 work item` / `関連改善 work item` を辿って復元する（例: `0001` の `[判断]03`〔ローカル git を残す〕→ `[判断]07`〔git 完全削除へ更新〕）。
- レビュー観点は「基準（`## レビュー観点`）」と「結果（`### 工程レビュー`）」を対にして示す。

### フェーズ4: CITE — 出典付きで回答 / 記録なしを明示

各主張に出典を添えて回答する。問いに答える証跡が見つからなければ「記録なし」と明示する。

## 問いの型 → 検索戦略

| 問い | 索引 | タグ / セクション |
|---|---|---|
| なぜこう実装した? | `index.md` → 該当 work item | `## 目的` / `## 背景` / `### [判断]` |
| 何を変えた? | `index.md` → 該当 work item | `## 受け入れ条件` / `### エージェント実行` |
| レビューで何を見る? | `index.md` → 該当 work item | `## レビュー観点` ＋ `### 工程レビュー` |
| この方針の経緯は? | `index.md` / `feedback/README.md` | `### [判断]` 連鎖 ＋ `出典 work item` を逆引き |
| この規約はどこ由来? | `feedback/README.md` | `## F<NN>` の `出典 work item` |

## 出力フォーマット

```
## 問い: [元の問い]

### 回答
[要約した答え]

### 根拠（出典引用）
- [主張1] — `issues/000N-*.md ## 目的`
- [主張2] — `issues/000N-*.md ## 人間の判断・フィードバック [判断]NN`

### 確度
記録あり | 部分的（一部は未記録）| 記録なし
```

## 停止条件

- 問いに答える証跡が揃った（出典を提示できる）。
- または、索引・タグを一通り当たっても該当記録が無いと確認できた（「記録なし」を返す）。

無理に `LOOP` を続けず、`$iterative-retrieval` と同様に絞り込みは数サイクルで打ち切る。

## 関連

- `$local-memory` — 検索対象の work item / 実行ログ / 横断フィードバックのフォーマット定義。
- `$iterative-retrieval` — フェーズ構造（DISPATCH→EVALUATE→REFINE→LOOP）の土台。
- `memory-searcher` — このスキルをプリロードし、controller の問いに引用付きで答えるループ外補助エージェント。
